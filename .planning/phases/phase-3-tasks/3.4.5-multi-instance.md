# Task 3.4.5 — Multi-instance layout + `--name` flag everywhere

**Status:** ⏳ pending
**Estimated effort:** ~1.5 hr
**Depends on:** 3.4.3 (engine), 3.4.4 (manifest)
**Blocks:** 3.4.6 (personal command needs the instance name), 3.4.7 (migration)

## Goal

Move from the single-instance layout (`~/.claudify/...`) to the
multi-instance layout (`~/.claudify/instances/<name>/...`). Every
entrypoint accepts `--name <NAME>` (default: `default`). systemd
service per instance: `claudify-<name>.service`.

## Why

`docs/architecture.md §3b` describes the runtime layout. `§5a`
defines the registry. Multi-instance is a core invariant of the
architecture; until paths nest under `instances/<name>/`, the rest
is documentation without code.

## Scope

### Files to modify

- `lib/configs.sh` — paths become `~/.claudify/instances/$INSTANCE_NAME/...`
- `lib/oauth.sh` — same
- `lib/service.sh` — unit name becomes `claudify-$INSTANCE_NAME.service`; `WorkingDirectory` per-instance
- `lib/onboarding.sh` — `collect_inputs` adds an instance-name prompt (or uses env `INSTANCE_NAME`); validates per `§3d`
- `lib/validate.sh` — add `validate_instance_name` (regex + blocklist of common Unix commands)
- `lib/args.sh` — add `--name` flag to install
- `lib/manifest.sh` — already path-aware from 3.4.4; double-check it uses the new paths
- `install.sh`, `update.sh`, `uninstall.sh`, `doctor.sh` — all gain `--name` flag (default: `default`)
- `lib/preflight.sh` — `preflight_linger` is per-user not per-instance; no change. Other checks may need to be per-instance.
- `build.sh` — no change (already concatenates lib/)
- `lib/README.md` — update function table for any changed signatures

### Files to create

- `tests/bash/multi-instance.bats` — install two instances side-by-side; confirm independence

### Files to delete

- none

## Key design points (from architecture.md §3b)

- Default instance name: `default`
- Path pattern: `~/.claudify/instances/<name>/`
- Service name: `claudify-<name>.service`
- Working dir: `~/.claudify/instances/<name>/workspace`
- Manifest: `~/.claudify/instances/<name>/claudify.json`
- Registry update: `manifest_register_instance <name>`

## Steps

1. Add `INSTANCE_NAME` global (set by `parse_args`, default `default`)
2. Define `claudify_instance_dir()` helper in `lib/manifest.sh`: returns `$HOME/.claudify/instances/$INSTANCE_NAME`
3. Replace every hard-coded path in `lib/configs.sh`, `lib/oauth.sh`, `lib/service.sh`, `lib/onboarding.sh` (where it touches `~/.claudify/...`) with `$(claudify_instance_dir)/...`
4. Service unit name: `claudify-$INSTANCE_NAME.service` everywhere it's referenced (`write_service`, `start_service`, `final_summary` if it shows the unit name)
5. Add `validate_instance_name` to `lib/validate.sh`:
   - Regex: `^[a-z][a-z0-9_-]{1,30}$`
   - Blocklist (env BLOCKLIST or hard-coded array): `ls cd cp mv rm cat grep find git npm bun node claude claudify docker systemctl journalctl sudo bash sh zsh`
6. `lib/args.sh` — add `--name` parsing; validate via `validate_instance_name`
7. `install.sh`, `update.sh`, `uninstall.sh`, `doctor.sh` — accept and forward `--name` (uninstall: `--name` becomes *required* unless `--all` is passed)
8. `doctor.sh` without `--name` — reads registry, shows a one-line summary per instance
9. Add `tests/bash/multi-instance.bats`:
   - Install instance `alpha` with fake env vars + `--non-interactive`
   - Install instance `beta` with different fake env vars
   - Confirm both `~/.claudify/instances/alpha/` and `.../beta/` exist with separate state
   - Confirm two systemd units; both startable; neither stomps the other
   - (Skip cleanly on systems without bats)
10. `bash test.sh` — green
11. **Station11 round-trip:**
    - `bash ~/test/uninstall.sh --name default --yes`
    - `bash ~/test/autoinstall.sh` — installs as `default` by default
    - `bash ~/test/doctor.sh` (no flag) — reports `default: ✓`
    - `bash ~/test/doctor.sh --name default` — same, but explicit
    - Bonus: `bash ~/test/autoinstall.sh` with a different `--name` would create a second instance — for now just confirm the first one works
12. CHANGELOG: `### Added — multi-instance support: every install is named (default: 'default'), runs under its own systemd unit (claudify-<name>.service), and lives in ~/.claudify/instances/<name>/.`

## Notes / risks

- The `default` default name is important. Don't make `--name` required for install — that's a UX regression for single-instance users. Required only for uninstall (where ambiguity is dangerous).
- Two instances using the same Telegram bot token is a runtime conflict (Telegram API allows only one polling client per token). Validate at install: refuse if the bot token is already in another instance's `.env`.
- The `claudify-<name>.service` naming change is breaking for existing installs. 3.4.7 (migration) is what handles that. Until 3.4.7 lands, **the new install layout is incompatible with the old layout**. Don't deploy 3.4.5 alone to Station11; bundle 3.4.5 + 3.4.7 in the same release.
