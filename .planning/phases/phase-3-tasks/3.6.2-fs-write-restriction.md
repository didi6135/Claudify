# Task 3.6.2 — Filesystem write-restriction (`ProtectHome` + `ReadWritePaths`)

**Status:** ⏳ pending
**Estimated effort:** ~45 min
**Depends on:** 3.6.1 (Tier-1 already in place), 3.5 (paths stable)
**Blocks:** 3.6.3 (syscall filter is the next layer)
**Research:** [`.planning/research/security.md`](../../research/security.md) Part C §2

## Goal

After this task, the bot can only **write** to four explicit paths:
`~/.claudify`, `~/.claude`, `~/.npm-global`, `~/.bun`. Everything
else under `$HOME` is read-only to the bot. A prompt-injected
`rm -rf ~/.ssh` silently fails. A skill's runaway `>>` into
`~/.bashrc` errors out.

This is the **single biggest security win** in the Phase 3.6
umbrella.

## Why

Today, a compromised conversation or buggy skill has full write
access to the operator's entire home directory. With `bypassPermissions`
in effect (no per-tool prompts), there is nothing between the LLM's
decision to write a file and the file being written. Filesystem
write-restriction at the systemd layer is the right place to bound
this.

See `.planning/research/security.md` Part C §2 for rationale.

## Scope

### Files to modify

- `lib/service.sh` — `write_service` heredoc gains `ProtectHome` +
  `ReadWritePaths`
- (potentially) `lib/engines/claude-code.sh::_npm_prefix_setup` —
  if we move PATH wiring out of `~/.bashrc` (see "decisions")
- `CHANGELOG.md` — `### Security` entry

### Files to create

- none

## The block

Adds to the `[Service]` section, after the Tier-1 directives from 3.6.1:

```ini
# === Filesystem write-restriction ===
# $HOME is read-only by default; everything else listed here is
# explicitly read+write. Anything outside this list is read-only or
# (for /etc, /usr) untouched by us anyway. A prompt-injected
# rm -rf ~/.ssh silently fails because ~/.ssh is read-only.
ProtectHome=read-only
ReadWritePaths=%h/.claudify %h/.claude %h/.npm-global %h/.bun
```

### Why each path is on the list

| Path | Why |
|---|---|
| `%h/.claudify` | Our state — bot has to write configs, manifests, conversation log, persona DB, etc. |
| `%h/.claude` | Claude Code's own state (`~/.claude.json`, `~/.claude/settings.json`, plugin cache). The engine writes here on every session. |
| `%h/.npm-global` | Where the `claude` binary lives + npm-installed plugins. Claude installs/updates plugins by writing here. |
| `%h/.bun` | Bun runtime. Bun caches build artefacts here. |

If you add a path the bot doesn't need write access to, the bot
gets quieter mystery failures. The list above is the minimum that
keeps Claude Code, Bun, and our state working.

### What about `~/.bashrc`?

Today's `_npm_prefix_setup` appends a `PATH` export line to
`~/.bashrc`:

```bash
echo "export PATH=\"$NPM_PREFIX/bin:\$PATH\"" >> "$rc"
```

**That's at install time, not at runtime.** The bot itself never
writes to `.bashrc` — only `install.sh` does, and `install.sh` runs
as the operator's interactive shell, NOT under the systemd unit.
So `ProtectHome=read-only` doesn't affect this code path.

We do NOT need to add `~/.bashrc` to `ReadWritePaths`.

## Decisions

### `ProtectHome=read-only` vs `=tmpfs` vs `=true`

| Option | What it does | Choice |
|---|---|---|
| `ProtectHome=true` | Hides $HOME entirely (empty mount) — bot can't see anything in $HOME | ❌ Bot needs to read CLAUDE.md, persona, etc. |
| `ProtectHome=read-only` | Bot can read $HOME but can't write anywhere except `ReadWritePaths` | ✅ This is what we want |
| `ProtectHome=tmpfs` | $HOME replaced with empty tmpfs — same as `=true` for our purpose | ❌ |

### Why not `ProtectSystem=strict`?

`ProtectSystem=strict` makes `/usr`, `/boot`, `/etc`, and other
system dirs read-only. Tempting, BUT: the npm-global symlinks for
`claude` go through `/usr/local/bin/claude` on some setups. If
`/usr` is unwritable and we ever try a global claude reinstall via
the bot, it'd fail. Tier-1's `ProtectKernelTunables` +
`ProtectControlGroups` + `ProtectClock` + `ProtectHostname` cover
most of `ProtectSystem`'s value without the foot-gun.

We could revisit `ProtectSystem=full` (which excludes `/etc`) in
Phase 5 if a hardening guide identifies it as low-risk for our
shape.

## Steps

1. Edit `lib/service.sh::write_service` — add the
   `ProtectHome=read-only` + `ReadWritePaths=` block after the
   Tier-1 block (3.6.1).
2. **Sanity-check no installer code path writes outside the
   listed paths.** `grep -rn 'echo.*>>.*\$HOME\|>> *~\|echo.*>>.*%h' lib/`.
   Anything matching needs to be either inside `~/.claudify`,
   `~/.claude`, `~/.npm-global`, `~/.bun`, OR run only at install
   time outside the unit (which is fine).
3. `bash build.sh` — `dist/install.sh` rebuilds.
4. `bash -n` clean.
5. **Station11 round-trip:**
   - Clean install
   - Service starts cleanly
   - **Test the protection actually works:**
     ```bash
     ssh ... 'systemd-run --user --pty --service-type=oneshot --unit=test-write \
        --property=ProtectHome=read-only \
        --property=ReadWritePaths=%h/.claudify \
        /bin/bash -c "echo X > ~/.testfile" 2>&1'
     # → should fail with "Read-only file system"
     ```
   - Bot replies to a real Telegram message
   - `bash ~/test/doctor.sh` 28/28
   - `systemd-analyze --user security` ≤ 3.0
6. CHANGELOG: `### Security — Bot service is now filesystem-restricted via ProtectHome=read-only + ReadWritePaths=%h/.claudify %h/.claude %h/.npm-global %h/.bun. Anything else under $HOME is read-only to the bot. systemd-analyze score: 5.0 → ~3.0.`

## Notes / risks

- **Biggest risk: a path the bot needs to write that's not on the
  list.** Mitigation: Station11 round-trip with full reply test
  catches this immediately (the bot will fail loudly, not silently).
  If we hit this in production, adding to `ReadWritePaths` is a
  one-line fix that re-runs.
- **Future engines may have different write needs.** A Gemini
  adapter might write to `~/.config/gemini/`. The list will need to
  grow per-adapter. Could move it to `engine_run_args` output (each
  adapter contributes its own `ReadWritePaths`) — but that's a 3.4.x
  refactor not in scope here.
- **MCP subprocesses inherit this restriction.** Future
  `claudify-memory` MCP (Phase 4.0a) writes to
  `~/.claudify/instances/<n>/data/` — already covered by
  `~/.claudify`. Future skills need to write to their own data dirs
  — same coverage.
- **`%h` is systemd's expansion for $HOME** — works for user-mode
  units. Don't write `$HOME` literally; systemd doesn't expand it.
