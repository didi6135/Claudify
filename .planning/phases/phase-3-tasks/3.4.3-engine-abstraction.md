## Goal

Extract everything Claude-Code-specific into `lib/engines/claude-code.sh`,
implementing the 6-function adapter contract from `architecture.md ¬ß6`.
The rest of `lib/` stops referring to `claude` directly and instead
calls abstract `engine_*` functions. Today there's one adapter; future
adapters (Gemini, OpenAI) will plug in without touching anything else.

## Why

`docs/architecture.md ¬ß6` defines the contract. ADR 0005 commits us to
*"engine-agnostic in design, Claude-only in implementation"*. This task
is what makes the design real ‚Äî separates harness code from engine
code.

## The contract

Every `lib/engines/<id>.sh` MUST export these functions:

| Function | Args | Behavior |
|---|---|---|
| `engine_install` | ‚Äî | Install the engine binary (e.g., `npm install -g @anthropic-ai/claude-code`). Idempotent ‚Äî skip if present. |
| `engine_auth_check` | ‚Äî | Returns 0 if currently authenticated, non-zero otherwise |
| `engine_auth_setup` | ‚Äî | Run interactive auth flow; persist credentials to `$CREDS_FILE` |
| `engine_run_args` | ‚Äî | Echo the argv vector for systemd `ExecStart` (channels, plugins, mode flags) |
| `engine_status` | ‚Äî | Echo a JSON object: `{"engine":"...","version":"...","authenticated":true/false}` |
| `engine_uninstall` | ‚Äî | Remove engine-specific state from `~/.claudify/instances/<name>/`. Does NOT remove the engine binary. |

Plus optional helpers (each engine adapter can define its own private
funcs, but the public API is exactly these 6).

## Steps

1. Create `lib/engines/claude-code.sh`. Add file header per `conventions.md`.
2. Implement the 6 functions:
   - `engine_install` ‚Üê move logic from `setup_npm_prefix` + `install_claude` (3.4.2's `claude.sh`)
   - `engine_auth_check` ‚Üê rename `claude_is_authed`
   - `engine_auth_setup` ‚Üê extract the OAuth-token-capture from `oauth_setup`. Note: the *interactive prompt logic* stays in `lib/oauth.sh` (UI is engine-agnostic); the engine adapter only knows how to drive `claude setup-token`.
   - `engine_run_args` ‚Üê echoes: `-qfec "claude --permission-mode bypassPermissions --channels plugin:telegram@claude-plugins-official" /dev/null`
   - `engine_status` ‚Üê parse `claude auth status` JSON + `claude --version`
   - `engine_uninstall` ‚Üê currently a no-op (engine binary is shared across instances)
3. Create `lib/engine.sh` as a thin dispatcher:
   - Reads `CLAUDIFY_ENGINE` env var (default: `claude-code`)
   - `source` the right adapter based on it
   - Define one tiny helper `engine_id()` that returns the current engine name
4. Update `lib/oauth.sh` to call `engine_auth_*` instead of `claude_*`
5. Update `lib/configs.sh` plugin install to use `engine_install_channel_plugin <name>` (add this 7th optional func to the contract ‚Äî it's specific to channel-plugin engines like Claude Code)
6. Update `lib/service.sh` `write_service`: ExecStart line becomes `/usr/bin/script $(engine_run_args)` (or similar ‚Äî adapt to keep the script invocation right)
7. Delete `lib/claude.sh`
8. Update `install.sh` source order ‚Äî replace `claude.sh` with `engine.sh`
9. Update `build.sh` MODULES ‚Äî drop `claude.sh`, add `engine.sh`. Bundle `lib/engines/claude-code.sh` into dist (it's part of the runtime).
10. `bash -n` everything, `bash build.sh`, every file ‚â§ 300 lines
11. **Station11 round-trip:** `bash ~/test/update.sh` ‚Äî installs, service starts, bot replies
12. `bash ~/test/doctor.sh` ‚Äî 28/28 green
13. CHANGELOG: `### Changed ‚Äî extracted engine-specific logic to lib/engines/claude-code.sh; lib/* now uses the engine_* abstraction. Adding a second engine in the future is a single new file.`

## Notes / risks

- The 6-function contract is the API stability surface. Once defined, future engines depend on it not changing. Document precisely.
- `engine_auth_setup` interactivity is the trickiest piece. Today's flow shows a URL, waits for the user to paste a code, then captures the token from the install log. Keep that flow intact ‚Äî only abstract the "talk to claude" part.
- Watch for env-variable leakage: `CLAUDE_CODE_OAUTH_TOKEN` is engine-specific; don't surface it as `ENGINE_TOKEN` in the abstraction. Each adapter manages its own env-var names internally.
